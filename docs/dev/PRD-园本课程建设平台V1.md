## 一、产品概述

### 1.1 产品定位

面向**全国幼儿园**的园本课程建设 SaaS 平台。通过 AI 驻园专家（9 步 Agent 流程），帮助幼儿园从「输入园所信息」一键生成「完整课程体系」，实现课程建设的专业化、标准化和个性化。

### 1.2 产品愿景

> 让每一所幼儿园都能拥有专业、科学、可落地的园本课程体系。

### 1.3 核心价值

| 价值维度 | 说明 |
|---------|------|
| **对园长** | 降低课程建设门槛，无需聘请专家团队，AI 即可生成符合政策要求、体现园所特色的完整课程体系 |
| **对教师** | 一键生成周教学计划、主题资源包，减少备课负担，聚焦教学本身 |
| **对园所** | 沉淀课程资产和实施记录，形成可追溯、可复用的园本课程资源库 |

### 1.4 产品口号

> AI 驻园专家 · 引领园本课程建设全流程

### 1.5 产品形态

- **类型**：Web SaaS 平台（浏览器访问）
- **部署方式**：云服务器部署，按园所开通账号
- **V1 技术栈**：Python + FastAPI（后端）、Vue 3 + Vite（前端）、PostgreSQL（数据库）、DeepSeek API（AI）

## 二、用户与角色

### 2.1 角色定义

| 角色 | 人数占比 | 典型画像 | 核心诉求 |
|------|---------|---------|---------|
| **园长** | 1人/园 | 30-50岁，负责园所整体运营和课程方向把控 | 快速生成专业课程体系，用于示范园评定、课程评比 |
| **教师** | N人/园 | 22-45岁，一线带班教师 | 基于课程方案快速生成教案和资源包，减轻备课负担 |

### 2.2 角色权限矩阵

| 功能 | 园长 | 教师 |
|------|:----:|:----:|
| 注册账号 | ✅ | ❌（由园长创建） |
| 登录/登出 | ✅ | ✅ |
| 📋 园本课程建设（填写信息 + 9步AI流程） | ✅ | ❌ |
| 生成园本课程方案 | ✅ | ❌ |
| 📝 课程生成（周计划/资源包） | ✅ | ✅ |
| 📂 我的园本课程（查看/编辑/下载方案） | ✅ | ✅ |
| 上传实施记录 | ✅ | ✅ |
| 👥 账号管理（创建/删除教师账号） | ✅ | ❌ |

### 2.3 账号体系规则

| 规则 | 说明 |
|------|------|
| 园长注册 | 填写「园所名称 + 姓名 + 手机号 + 密码」→ 自动创建园所 |
| 教师创建 | 园长在「账号管理」中填写「教师姓名 + 手机号 + 初始密码」→ 自动绑定到当前园所 |
| 数据隔离 | 同一园所内的园长和教师共享该园所的所有方案、课程、记录；不同园所之间数据完全隔离 |
| 手机号唯一 | 同一手机号不可重复注册或创建 |
| 密码要求 | 至少 6 位 |

## 三、用户场景与故事

### 3.1 园长 — 建设课程

> **场景**：张园长正在准备省级示范园评定，需要一套完整的园本课程体系。

| 步骤 | 用户行为 | 系统响应 |
|------|---------|---------|
| 1 | 张园长注册账号，填写园所名称「北京市XX幼儿园」 | 创建园所和园长账号，进入主应用 |
| 2 | 进入「园本课程建设」，填写园所信息（特色：自然教育；资源：柚子树、菜园；目标：健康儿童、探究儿童） | 表单实时校验，进度条更新 |
| 3 | 点击「提交信息，启动AI诊断」 | 依次展示 9 个 AI 专家的分析结果（园所诊断→政策研究→…→评审委员会） |
| 4 | 查看报告汇总页，点击「生成园本课程方案」 | 生成完整课程方案（含课程起源、理念、目标、内容、实施、评价、保障七大章节），保存到「我的园本课程」 |
| 5 | 在「我的园本课程」中查看方案，下载 PDF 提交评审 | 方案可预览、编辑、下载、打印 |

### 3.2 教师 — 生成课程

> **场景**：李老师（大班）下周要开展「我身边的运动」主题活动，需要教案和资源包。

| 步骤 | 用户行为 | 系统响应 |
|------|---------|---------|
| 1 | 李老师用园长分配的账号登录 | 进入主应用，默认展示「课程生成」版块 |
| 2 | 选择园本课程方案「活力童年」，输入主题「我身边的运动」，选择大班 | 系统加载方案上下文 |
| 3 | 选择「周教学计划」，指定第 1 周，点击生成 | 生成周一至周五每日活动安排表（含目标、材料、流程、延伸建议） |
| 4 | 预览满意，点击「保存到我的园本课程」 | 保存到方案关联的课程内容中 |
| 5 | 课后上传幼儿作品照片和观察记录 | 实施记录按主题归档，可筛选查看 |

### 3.3 园长 — 管理教师

> **场景**：张园长新招了 3 位老师，需要给她们开通平台账号。

| 步骤 | 用户行为 | 系统响应 |
|------|---------|---------|
| 1 | 点击右上角用户名 → 「账号管理」 | 弹出教师账号管理弹窗 |
| 2 | 填写教师姓名、手机号、初始密码，点击「添加」 | 创建教师账号，自动绑定到本园 |
| 3 | 将账号信息告知教师 | 教师可用该账号登录，看到本园的课程方案 |

## 五、功能详细设计

---

### 5.1 账号系统

#### 5.1.1 园长注册

**入口**：欢迎页 → 点击「进入平台」→ 点击「园长注册」

**输入字段**：

| 字段 | 类型 | 必填 | 校验规则 | 示例 |
|------|------|:----:|---------|------|
| 园所名称 | 文本输入 | ✅ | 2-50字，不可为空 | 北京市阳光花园幼儿园 |
| 园长姓名 | 文本输入 | ✅ | 2-20字 | 张老师 |
| 手机号 | 文本输入 | ✅ | 11位手机号格式，不可与已有账号重复 | 13800138000 |
| 密码 | 密码输入 | ✅ | 至少6位 | 123456 |

**业务规则**：
- 注册成功后，自动创建园所记录，并登录跳转到主应用
- 手机号已存在时提示「该手机号已注册」，引导登录
- 密码在前端做基础校验，后端做加密存储

**输出**：创建园所 + 园长账号 → 自动登录 → 跳转主应用

---

#### 5.1.2 登录

**入口**：欢迎页 → 点击「进入平台」

**输入字段**：

| 字段 | 类型 | 必填 | 示例 |
|------|------|:----:|------|
| 手机号 | 文本输入 | ✅ | 13800138000 |
| 密码 | 密码输入 | ✅ | 123456 |

**业务规则**：
- 手机号不存在时提示「账号不存在」
- 密码错误时提示「密码错误」
- 登录成功后根据角色跳转：园长 → 园本课程建设；教师 → 课程生成
- 登录态通过 JWT Token 维持，有效期 2 小时

---

#### 5.1.3 账号管理（园长专属）

**入口**：右上角用户名下拉菜单 → 「账号管理」

**功能**：
- 查看本园所有教师账号列表（姓名、手机号）
- 添加教师账号
- 删除教师账号

**添加教师 — 输入字段**：

| 字段 | 类型 | 必填 | 校验规则 | 示例 |
|------|------|:----:|---------|------|
| 教师姓名 | 文本输入 | ✅ | 2-20字 | 李老师 |
| 手机号 | 文本输入 | ✅ | 11位，不可与本园已有账号重复 | 13900139000 |
| 初始密码 | 文本输入 | ✅ | 至少6位 | ab1234 |

**业务规则**：
- 删除教师账号时，该教师已上传的实施记录保留（归属于园所，不随账号删除）
- 手机号已被本园其他教师使用时提示「该手机号已存在」

### 5.2 版块一：园本课程建设

#### 5.2.1 园所信息录入

**入口**：主应用 → 「园本课程建设」Tab → 默认展示录入表单

**页面布局**：左侧步骤导航（11步）+ 右侧主内容区（表单）

**输入模块**（6大模块）：

**模块一：园所基本信息**

| 字段 | 类型 | 必填 | 示例 |
|------|------|:----:|------|
| 园所名称 | 文本输入 | ✅ | 北京市阳光花园幼儿园 |
| 所在地 | 文本输入 | — | 北京市朝阳区 |

| 字段 | 类型 | 示例 |

**模块三：地域资源**

| 字段 | 类型 | 示例 |
|------|------|------|

**模块四：培养目标**

| 字段 | 类型 | 示例 |
|------|------|------|
| 培养目标 | 文本域（自由输入） | 健康儿童、自主儿童、探究儿童、创造儿童、友善儿童 |

> 💡 提示：请输入园所希望培养的儿童关键品质，多个请用逗号或顿号分隔

**模块五：未来规划**

| 字段 | 类型 | 示例 |

**模块六：微观环境补充**

| 子模块 | 字段 | 类型 | 示例 |
|--------|------|------|------|
| （1）园内微环境 | 园内环境特征 | 文本域 | 菜园、沙地、水池、坡地、柚子树、桂花树、塑胶操场 |
| （2）周边300米资源 | 周边社区资源 | 文本域 | 社区菜市场、公园、河道、商铺、博物馆、消防站、花店、图书馆 |
| （3）教师行为画像 | 教师风格描述 | 文本域 | （表达型）→ 课程偏向故事化、角色扮演；（科学型）→ 课程偏向实验、探究 |

**业务规则**：
- 仅「园所名称」为必填，其余选填，但填写越详细生成越精准
- 页面顶部显示表单完成进度条
- 点击「提交信息，启动AI诊断」后进入 9 步 Agent 流程

---

#### 5.2.2 九步 AI 专家流程

用户提交信息后，系统按顺序展示 9 个 Agent 的分析结果。每一步包含：
- **专家卡片**：头像/图标 + 专家名称 + 角色描述
- **分析内容**：结构化输出（标题 + 正文 + 表格/列表）
- **交互操作**：确认/采纳/下一步

| 步骤 | Agent 名称 | 角色 | 核心输出 |
|:----:|-----------|------|---------|
| 1 | 园所诊断师 | 课程现状分析专家 | 优势/问题/机会/风险分析、SWOT分析、课程成熟度等级 |
| 2 | 政策研究员 | 政策合规专家 | 课程建设要求、改革方向、政策关键词（游戏化/生活化/项目化/儿童本位/家园共育）、建设建议 |
| 3 | 地域文化挖掘师 | 文化资源专家 | 自然资源/文化资源/产业资源/社区资源/非遗资源梳理、课程转化建议 |
| 4 | 特色定位专家组 | 4位专家竞争 → 裁判择优 | **专家A**（教育学派）：生活/游戏/探究课程方向<br>**专家B**（品牌策划）：课程名称/宣传口号/招生亮点<br>**专家C**（课程专家）：课程特色/价值<br>**专家D**（儿童发展）：儿童成长画像<br>**裁判Agent**：从4方案中选最优 → 输出课程定位/名称/主张/口号 |
| 5 | 课程架构师 | 3位专家协同 | **专家A**（教育学）：课程理念<br>**专家B**（课程）：课程目标<br>**专家C**（设计）：课程结构/模块<br>→ 输出课程理念/目标/框架/模块 |
| 6 | 课程地图设计师 | 进阶体系设计 | 小班（感知体验）→ 中班（探索发现）→ 大班（创造表达）完整课程地图 |
| 7 | 课程实施专家组 | 4位专家 | **专家A**：环境创设<br>**专家B**：一日生活<br>**专家C**：区域活动<br>**专家D**：家园共育<br>→ 输出《课程实施方案》 |
| 8 | 课程评价专家组 | 4位专家 | **幼儿评价**：儿童发展评价<br>**教师评价**：教师成长评价<br>**课程评价**：课程质量评价<br>**家长评价**：家园共育评价<br>→ 输出完整评价体系（含评价表） |
| 9 | 课程评审委员会 | 模拟外部评审 | 模拟教育局评审专家/课程比赛评委/示范园评审专家/集团园课程专家<br>→ 打分：课程先进性/逻辑性/特色性/落地性/政策契合度<br>→ 输出修改建议 → 回流优化 |

**业务规则**：
- 步骤必须按顺序进行，前一步完成后才能进入下一步
- 每一步展示时带有打字机动画效果
- 每步内容基于用户输入的园所信息动态生成（V1 为模板 + 变量替换，后续版本接入 AI API）

#### 5.2.4 一键生成课程方案

**触发**：报告汇总页 → 点击「生成园本课程方案」

**生成流程**：
```
用户点击 → 模板即时渲染（< 100ms）→ 显示状态条「✨ AI 正在优化方案...」
→ 后台调用 DeepSeek API（10-30s）→ API 返回后替换为 AI 优化版本
→ 状态条变为「✅ AI 优化完成」→ 自动保存到「我的园本课程」→ 跳转
```

**方案文档结构**（七大章节）：

| 章节 | 内容要求 |
|------|---------|
| 一、课程起源 | 基于园所特色、地域资源、微观环境，阐述课程产生的背景和必然性 |
| 二、课程理念 | 凝练核心教育主张，体现园所哲学 |

**关键约束**：

## 八、数据模型

### 8.1 实体关系

```
园所（kinder） 1 ──── N 账号（account）
园所（kinder） 1 ──── N 课程方案（curriculum_plan）
课程方案（curriculum_plan） 1 ──── N 课程内容（course_content）
课程方案（curriculum_plan） 1 ──── N 实施记录（implementation_record）
```

### 8.2 数据表设计

#### 园所表（kinders）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| name | VARCHAR(100) | 园所名称 |
| created_at | TIMESTAMP | 创建时间 |

#### 账号表（accounts）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| role | ENUM('principal','teacher') | 角色 |
| name | VARCHAR(50) | 姓名 |
| phone | VARCHAR(20) UNIQUE | 手机号 |
| password_hash | VARCHAR(255) | 密码哈希 |
| kinder_id | BIGINT FK | 所属园所 |
| created_at | TIMESTAMP | 创建时间 |

#### 课程方案表（curriculum_plans）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| kinder_id | BIGINT FK | 所属园所 |
| name | VARCHAR(200) | 方案名称 |
| feature_word | VARCHAR(50) | 核心特色词 |
| resource_word | VARCHAR(50) | 地域资源词 |
| goal_word | VARCHAR(50) | 培养目标词 |
| form_data | JSONB | 原始表单数据 |
| html_content | TEXT | 生成的完整方案 HTML |
| created_by | BIGINT FK | 创建人（园长） |
| created_at | TIMESTAMP | 创建时间 |
| updated_at | TIMESTAMP | 更新时间 |

#### 课程内容表（course_contents）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| plan_id | BIGINT FK | 关联方案 |
| kinder_id | BIGINT FK | 所属园所 |
| theme | VARCHAR(100) | 主题名称 |
| age_group | VARCHAR(20) | 年龄段（小班/中班/大班） |
| type | ENUM('weekly_plan','resource_pack') | 类型 |
| content | JSONB | 结构化内容 |
| html_content | TEXT | 渲染后的 HTML |
| created_by | BIGINT FK | 创建人 |
| created_at | TIMESTAMP | 创建时间 |
| updated_at | TIMESTAMP | 更新时间 |

#### 实施记录表（implementation_records）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| plan_id | BIGINT FK | 关联方案 |
| kinder_id | BIGINT FK | 所属园所 |

| title | VARCHAR(200) | 标题 |

| file_url | VARCHAR(500) | 文件存储地址（V1 为 Base64，V2 为 OSS URL） |
| file_type | VARCHAR(50) | MIME 类型 |
| created_by | BIGINT FK | 上传人 |
| created_at | TIMESTAMP | 上传时间 |

### 8.3 数据隔离策略

- 所有业务表包含 `kinder_id` 字段
- PostgreSQL 行级安全策略（RLS）：每个请求只能访问自身 `kinder_id` 的数据
- JWT Token 中包含 `kinder_id`，每个 API 请求自动注入

## 九、非功能需求

### 9.1 性能

| 指标 | 要求 |
|------|------|
| 页面首屏加载 | < 3 秒 |

### 9.2 安全

| 要求 | 实现方式 |
|------|---------|
| API Key 保护 | 后端持有，前端不接触；通过 Python 代理层转发 |
| 密码存储 | bcrypt 哈希，不明文存储 |
| 鉴权 | JWT Token，有效期 2 小时 |
| 数据隔离 | PostgreSQL RLS，按 kinder_id 隔离 |
| 传输安全 | HTTPS（Let's Encrypt 免费证书） |
| 输入校验 | FastAPI Pydantic 模型自动校验，防止注入 |

### 9.3 兼容性

| 维度 | 要求 |
|------|------|
| 浏览器 | Chrome 90+、Edge 90+、Safari 14+、Firefox 90+ |
| 分辨率 | 最小支持 1024×768，推荐 1920×1080 |
| 移动端 | V1 不要求移动端适配，V2 考虑响应式 |

### 9.4 可用性

| 要求 | 指标 |
|------|------|
| 服务可用性 | 99.5%（月度） |
| 数据备份 | PostgreSQL 每日自动备份到 OSS，保留 7 天 |
| 故障恢复 | RTO < 4 小时，RPO < 24 小时 |

| Agent | AI 智能体，本产品中每个 Agent 扮演不同角色（诊断师、研究员等） |
| SaaS | 软件即服务，用户通过浏览器使用，无需安装 |
| RLS | 行级安全策略（Row-Level Security），PostgreSQL 的数据隔离机制 |
| 多租户 | 多个园所共用一套系统，但数据互相隔离 |
